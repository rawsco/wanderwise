# SCRUM-12 — Edit notes inline

## Context

Stops have a notes section: a list of free-text notes attached to a stop, with create + delete already implemented. Users can't currently edit a note — they have to delete and recreate it. This plan adds inline edit.

Notes are stored as a denormalized array on the Stop entity (`notes: StopNote[]`), keyed by `noteId`. CRUD goes through `src/app/api/trips/[tripId]/stops/[stopId]/notes/route.ts`. UI lives in `src/app/trips/[tripId]/stops/[stopId]/StopDetailClient.tsx` (the Booking tab → Notes card).

## Acceptance criteria (from ticket)

1. Edit option visible/accessible per note (edit icon).
2. Click → editable state, pre-populated with existing content.
3. User can modify text and save.
4. On save, note updated in place, displayed immediately.
5. User can cancel; original content unchanged.
6. Empty save (after trim) → validation message + save blocked.

## Design

### API — add PATCH to existing notes route

`src/app/api/trips/[tripId]/stops/[stopId]/notes/route.ts`:

```ts
const updateSchema = z.object({
  noteId: z.string().min(1),
  text: z.string().min(1).max(2000),
});

export async function PATCH(req, { params }) {
  await requireAuth();
  const { tripId, stopId } = await params;
  const { noteId, text } = updateSchema.parse(await req.json());

  const result = await StopEntity.get({ tripId, stopId }).go();
  if (!result.data) return NextResponse.json({ error: "Not found" }, { status: 404 });

  const existing: StopNote[] = (result.data.notes as StopNote[] | undefined) ?? [];
  const idx = existing.findIndex(n => n.noteId === noteId);
  if (idx === -1) return NextResponse.json({ error: "Note not found" }, { status: 404 });

  const updated: StopNote = { ...existing[idx], text };
  const next = [...existing.slice(0, idx), updated, ...existing.slice(idx + 1)];
  await StopEntity.update({ tripId, stopId }).set({ notes: next }).go();

  return NextResponse.json(updated);
}
```

Catch block mirrors the existing handlers: ZodError → 400, otherwise → 401.

Note: `createdAt` is **preserved**, not bumped — editing fixes the original note rather than creating a new one. Acceptance criterion 4 says "displayed immediately" but the spec doesn't ask for an `updatedAt` audit field, so I'm not adding one (no schema migration scope creep).

### UI — inline edit in `StopDetailClient.tsx`

State additions:
```ts
const [editingNoteId, setEditingNoteId] = useState<string | null>(null);
const [editText, setEditText] = useState("");
const [editSaving, setEditSaving] = useState(false);
const [editError, setEditError] = useState<string | null>(null);
```

Imports: add `Pencil`, `Check`, `X` from `lucide-react`.

Behavior:

- Read mode (default): show note text, timestamp, and two hover-revealed icon buttons — pencil (edit) + trash (delete, existing). Click pencil → `setEditingNoteId(note.noteId); setEditText(note.text); setEditError(null)`.
- Edit mode (when `editingNoteId === note.noteId`): replace the `<p>` text with a `<textarea>` (autoFocus, mirror styling of the add-note textarea), and replace the hover-button row with three always-visible buttons: Save, Cancel, plus the timestamp on the left. Below the textarea, show the validation error if `editError` is set.
- Save: trim text. If empty → `setEditError("Note can't be empty")` and return without calling the API. Otherwise PATCH to the notes route. On success: replace the note in `notes` state in place, exit edit mode. On non-OK response: keep editing, set a generic error.
- Cancel: reset edit state. No API call.
- Keyboard: `Cmd/Ctrl+Enter` saves (matching add-note pattern); `Esc` cancels.
- Disable Save button while `editSaving` or while trimmed text is empty.

Only one note may be in edit mode at a time (single `editingNoteId`).

### Out of scope

- No `updatedAt` field (would need entity schema work, not in ticket).
- No optimistic UI / undo — we wait for the server response, like the existing add-note flow.
- No edit history.
