# SCRUM-12 — Ability to edit notes

**Goal:** Let a user edit an existing stop note inline. Pencil icon next to the trash icon on each note; click pencil to enter an edit mode pre-populated with the current text; Save persists; Cancel leaves the note unchanged; empty save is blocked with a visible validation message.

**Ticket:** https://devotonomy.atlassian.net/browse/SCRUM-12
**Branch:** `scrum-12`

## Context for the implementer

**Notes today.** A `StopNote` is `{ noteId, text, createdAt }` (`src/types/stop.ts`). They're stored as an array on the stop's `notes` attribute (`src/lib/db/stop.entity.ts`, `type: "any"`). The notes UI lives in `src/app/trips/[tripId]/stops/[stopId]/StopDetailClient.tsx` inside the **Booking** tab. Each note renders with the text plus a hover-revealed `Trash2` button (line 289–304). The "add note" form is a raw `<textarea>` + `Button` below the list (line 309–329). State (`notes`, `noteText`, `saving`, `deleting`) sits on the client component.

**Existing API.** `src/app/api/trips/[tripId]/stops/[stopId]/notes/route.ts` exports two handlers — `POST` (create) and `DELETE` (delete by `noteId` in the body). Both follow the `requireAuth()` → Zod parse → ElectroDB `update().set({ notes: ... }).go()` pattern documented in `src/app/api/CLAUDE.md`. **DELETE takes the `noteId` in the request body**, not a path segment — so I'll match that convention with PATCH rather than introducing a new `[noteId]` dynamic route. Keeps the surface area smaller and consistent.

**Inline-edit precedent.** `src/components/stop/StopList.tsx` (around line 171–356, the `StopItem` component) is the canonical inline-edit pattern in this codebase: a `editing` boolean state, a pencil icon revealed on hover, a Save/Cancel button pair using `lucide-react`'s `Check`/`X`. We will follow that pattern's *spirit* (toggle between view/edit branches, lucide icons, ghost button for cancel) without lifting it wholesale, because notes are simpler — just one text field, no date inputs.

**Validation.** No automated tests in the repo. The validation gauntlet is `npx tsc --noEmit && npm run lint && npm run build`, plus visual confirmation via the worktree's dev stack at `https://192.168.50.155:3100` (the reviewer runs that — Phase 4d).

**Don't reformat unrelated code.** The whole change is one new handler in `notes/route.ts` (~30 lines) and one section of `StopDetailClient.tsx` (state additions + edit-mode JSX in the notes map). Touch nothing else.

## Task 1 — Add `PATCH` handler to the notes route

**Files:** Modify `src/app/api/trips/[tripId]/stops/[stopId]/notes/route.ts`.

- [ ] **Step 1.** Extend the Zod schemas at the top. Keep `createSchema` as-is. Add:

```ts
const updateSchema = z.object({
  noteId: z.string().min(1),
  text: z.string().min(1).max(2000),
});
```

The `text: z.string().min(1)` enforces the empty-content rule at the API boundary. The frontend will block empty saves before they leave the browser, but server-side validation is the real guard.

- [ ] **Step 2.** Append a `PATCH` handler after `DELETE`. It mirrors the existing handlers' shape — `requireAuth()` first, `params` awaited, body Zod-parsed, stop fetched, notes array mapped to replace the matching note's `text`, write back, return the updated note (so the client doesn't have to reconstruct it). 404 if the stop is missing **or** if the note ID is not found in the array — calling code shouldn't be allowed to mutate a non-existent note.

```ts
export async function PATCH(
  req: Request,
  { params }: { params: Promise<{ tripId: string; stopId: string }> }
) {
  try {
    await requireAuth();
    const { tripId, stopId } = await params;
    const { noteId, text } = updateSchema.parse(await req.json());

    const result = await StopEntity.get({ tripId, stopId }).go();
    if (!result.data) return NextResponse.json({ error: "Not found" }, { status: 404 });

    const existing: StopNote[] = (result.data.notes as StopNote[] | undefined) ?? [];
    const target = existing.find(n => n.noteId === noteId);
    if (!target) return NextResponse.json({ error: "Not found" }, { status: 404 });

    const updated: StopNote = { ...target, text };
    const nextNotes = existing.map(n => (n.noteId === noteId ? updated : n));
    await StopEntity.update({ tripId, stopId }).set({ notes: nextNotes }).go();

    return NextResponse.json(updated);
  } catch (err) {
    if (err instanceof z.ZodError) return NextResponse.json({ error: err.issues }, { status: 400 });
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }
}
```

We deliberately preserve `createdAt` (via the spread) — the note didn't get re-created, only edited. We do not add `updatedAt` since it isn't in the `StopNote` type and adding it would silently expand the public shape.

- [ ] **Step 2.** After the existing `deleting` state (around line 57), add four new pieces of state:


`editingId` is the noteId currently in edit mode (or null). `editText` is the in-progress draft. `editSaving` shows the spinner on the Save button. `editError` is the inline validation message ("Note cannot be empty") shown beneath the textarea when the user tries to save with empty content.

- [ ] **Step 3.** After `deleteNote` (around line 157), add three handler functions:

```tsx
function startEdit(note: StopNote) {
  setEditingId(note.noteId);
  setEditText(note.text);
  setEditError(null);
}

function cancelEdit() {
  setEditingId(null);
  setEditText("");
  setEditError(null);
}

async function saveEdit(noteId: string) {
  const text = editText.trim();
  if (!text) {
    setEditError("Note cannot be empty");
    return;
  }
  setEditSaving(true);
  setEditError(null);
  const res = await fetch(`/api/trips/${stop.tripId}/stops/${stop.stopId}/notes`, {
    method: "PATCH",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ noteId, text }),
  });
  if (res.ok) {
    const updated: StopNote = await res.json();
    setNotes(prev => prev.map(n => (n.noteId === noteId ? updated : n)));
    setEditingId(null);
    setEditText("");
  } else {
    setEditError("Could not save changes");
  }
  setEditSaving(false);
}
```

The optimistic-update style here matches `submitNote` (refetches the canonical note from the server response and merges by noteId). On API failure we surface the same inline `editError` slot rather than `alert()`-ing or silently failing — keeps the user in the edit form so they can retry without losing their text.

- [ ] **Step 4.** Replace the inner contents of the existing notes map (lines 288–305) so each note has both view and edit modes. The outer `<div className="group relative rounded-lg bg-gray-50 border border-gray-100 px-3 py-2.5">` stays. Inside, branch on `editingId === note.noteId`:

Notes on the patch:

- The pencil and trash buttons share a hover-reveal container (`opacity-0 group-hover:opacity-100`). On touch devices `group-hover` won't fire — that's a pre-existing limitation of the delete button and not something this ticket asks us to redesign. Touch users can still tap-and-hold or focus the area to reveal the actions if browser/OS conventions allow; deeper mobile UX is out of scope.
- `aria-label`s are added to both icon buttons. The existing trash button had no label; while we're touching this code we give both icons accessible names.
- The edit textarea uses `text-base` (per CLAUDE.md mobile rules — prevents iOS auto-zoom on focus). All inline color choices use Tailwind utility classes that are literal in the source (no dynamic class lookups), per the Tailwind v4 JIT rule in CLAUDE.md.
- `autoFocus` puts the cursor in the textarea when edit mode opens — saves a click on desktop, no cost on mobile.
- `Escape` to cancel and `Cmd/Ctrl+Enter` to save match the existing add-note form's keyboard affordance.

## Acceptance

Done when:

1. Each note in the Booking → Notes section shows a hover-revealed Pencil icon next to the existing Trash icon. Clicking the pencil swaps the note display for a textarea pre-populated with the current text plus Save/Cancel buttons.
2. Editing the textarea and clicking Save sends a `PATCH` to `/api/trips/{tripId}/stops/{stopId}/notes` with `{ noteId, text }`, replaces the note in local state with the response, and exits edit mode — the new text is visible immediately.
3. Clicking Cancel (or pressing Escape) exits edit mode without calling the API; the original note text remains unchanged.
4. Attempting to save with whitespace-only text shows "Note cannot be empty" beneath the textarea, keeps the user in edit mode, and does not call the API.
5. Server-side, `PATCH` requires auth, validates `{ noteId, text }` with Zod, returns 404 if the stop or noteId doesn't exist, and 400 on validation failure (including empty text).
6. `npx tsc --noEmit && npm run lint && npm run build` all pass on the branch.
