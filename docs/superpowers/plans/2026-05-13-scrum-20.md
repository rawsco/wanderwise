# SCRUM-20 — Delete Trip from My Trips page

**Goal:** Give every TripCard on `/trips` a touch- and mouse-friendly delete control that, after a confirmation prompt, removes the trip *and all its stops*.

**Ticket:** https://devotonomy.atlassian.net/browse/SCRUM-20
**Branch:** `scrum-20`

---

## Context for the implementer

The user already has a `DELETE /api/trips/[tripId]` route at `src/app/api/trips/[tripId]/route.ts:120` but it has two problems we have to fix on the way through:

1. **It does not cascade-delete stops.** It calls `TripEntity.delete({ userId, tripId }).go()` and stops at that — every `Stop` row for the trip is orphaned in DynamoDB. The ticket asks for "all trip information" to be deleted, so we must enumerate stops via `StopEntity.query.byTrip({ tripId })` and delete each before deleting the Trip itself.
2. **It does not validate ownership** the way `GET` and `PATCH` do (those query `byUser` filtered by `tripId` and return 404 if the row doesn't belong to the caller). A blind `delete({ userId, tripId })` against an ID that doesn't exist for the caller is silently a no-op but should be a 404 for the same reason `GET` / `PATCH` are.

Both are addressed in Task 1.

**UI placement — why the card itself, not the trip detail page.** The ticket literally says *"on the My Trips page"*, so the control belongs on each TripCard. The existing profile delete (`src/components/profile/DeleteProfileButton.tsx`) lives on the profile detail page, not on the list — useful as a code template but not as a placement model.

**Touch + mouse friendliness — no hover-only triggers.** Hover-reveal patterns (the button only appears when the mouse is over the card) are mouse-only by definition; on iOS Safari there is no hover state to enter. The delete control must therefore be **always visible**. We use a small ghost trash-icon button with a 40px square hit area (>= the iOS 44px target after padding) pinned at the top-right corner of each card.

**Confirmation prompt — native `confirm()` to match existing pattern.** `DeleteProfileButton` uses `window.confirm()` and there is no `<Dialog>` / `<AlertDialog>` component in `src/components/ui/`. Building a custom modal for one route would be premature abstraction; reuse `confirm()`. The wording mentions the trip name so a misaimed tap is recoverable.

**The hard layout problem — button inside a `<Link>` is invalid HTML.** The current TripCard wraps its whole body in a `<Link>`:

```tsx
<Link href={`/trips/${tripId}`}>
  <Card>…</Card>
</Link>
```

Putting an interactive `<button>` inside an `<a>` is invalid per HTML5 (interactive content cannot nest), and even with `e.preventDefault()` + `e.stopPropagation()` the underlying anchor element will still attract right-click / long-press menu actions that target the link, not the button. The clean fix is to restructure TripCard so the `<Link>` and the delete button are **siblings**, with the link covering the card body and the button laid over the top-right corner:

```tsx
<div className="relative h-full">
  <Link href={`/trips/${tripId}`} className="block h-full">
    <Card>…</Card>
  </Link>
  <DeleteTripButton … className="absolute top-2 right-2 z-10" />
</div>
```

This keeps the whole card clickable for navigation, keeps the delete button as a real `<button>`, and removes the need for `preventDefault` plumbing.

**No optimistic UI removal.** TripsPage (`src/app/trips/page.tsx`) is a server component that fetches data on render. After a successful `DELETE`, we call `router.refresh()` to re-fetch — same pattern `DeleteProfileButton` uses. The user briefly sees the card before it disappears on the next render tick; that is acceptable and matches profile delete UX.

**Validation — no test suite.** Per `AGENTS.md` the gauntlet is `npx tsc --noEmit && npm run lint && npm run build`. Browser smoke-test on the LAN dev URL is part of the hand-off, not part of plan validation.

## Task 1 — Fix `DELETE /api/trips/[tripId]` to cascade-delete and validate ownership

**Files:** Modify `src/app/api/trips/[tripId]/route.ts`.

The existing DELETE handler lives at lines 120-131 of the file. Replace it with a version that follows the same shape `GET` and `PATCH` use: query `byUser` filtered by `tripId`, return 404 if the trip is not the caller's, otherwise enumerate the stops and delete them in parallel, then delete the trip.

- [ ] **Step 1.** Replace the body of `export async function DELETE` so it:
  1. Calls `requireAuth()` inside `try` (first line) — already there.
  2. Awaits `params` for `tripId` — already there.
  3. Queries `TripEntity.query.byUser({ userId: user.id }).where(({ tripId: tid }, { eq }) => eq(tid, tripId)).go()`. If `.data[0]` is undefined, return `404` with `{ error: "Not found" }`.
  4. Queries `StopEntity.query.byTrip({ tripId }).go()` and `Promise.all`-deletes each via `StopEntity.delete({ tripId, stopId: stop.stopId }).go()`.
  5. Calls `TripEntity.delete({ userId: user.id, tripId }).go()`.
  6. Returns `NextResponse.json({ success: true })`.
- [ ] **Step 2.** Keep the outer `try / catch` shape exactly as the surrounding handlers use it — the `catch` returns `{ error: "Unauthorized" }` with status 401 so `requireAuth()` rejections still surface correctly. No new error branches are needed (no Zod schema, no body).
- [ ] **Step 3.** Sanity-check the diff against the `GET` handler at lines 25-44 — the ownership-check shape should be identical. If it differs (different `where` predicate, different 404 body), align it.

**Acceptance criteria:**
- A `DELETE` against a `tripId` belonging to another user returns `404 Not Found` (not 200, not 401).
- A `DELETE` against a `tripId` that has N stops deletes all N rows from the `Stop` partition before the trip row is removed (verifiable via DynamoDB Local on next run).
- An unauthenticated `DELETE` returns `401`.

## Task 2 — `DeleteTripButton` client component

**Files:** Create `src/components/trip/DeleteTripButton.tsx`.

- [ ] **Step 1.** New file, `"use client"` directive.
- [ ] **Step 2.** Props: `tripId: string`, `tripName: string`. Optional `className?: string` so the parent can position it.
- [ ] **Step 3.** Body:
  ```tsx
  const router = useRouter();
  async function handleDelete() {
    if (!confirm(`Delete "${tripName}"? This permanently removes the trip and all its stops.`)) return;
    await fetch(`/api/trips/${tripId}`, { method: "DELETE" });
    router.refresh();
  }
  ```
- [ ] **Step 4.** Render a `<button type="button" aria-label="Delete trip" onClick={handleDelete} …>` with a `<Trash2 className="h-4 w-4" />` icon from `lucide-react`. Do not use the `<Button>` UI primitive with `variant="destructive"` — that primitive is meant for primary destructive CTAs (full-bleed, clearly labelled). Here we need a *subtle* always-visible affordance that does not dominate the card. Use a bare `<button>` with Tailwind classes:
  - `h-9 w-9` for the 36px box + `inline-flex items-center justify-center rounded-md` for the visual shape.
  - `text-gray-400 hover:text-red-600 hover:bg-red-50 active:bg-red-100` for the interaction states.
  - `transition-colors`.
  - `cursor-pointer`.
- [ ] **Step 5.** Spread `className` onto the button so the parent can pass `absolute top-2 right-2 z-10`.
- [ ] **Step 6.** **No `preventDefault`/`stopPropagation`** — because the button will live *outside* the `<Link>` (see Task 3), the link click never fires when the user taps the button. Plumbing those guards in would be cargo-culted from blog posts about button-in-link.

**Acceptance criteria:**
- Component compiles and lints clean.
- Imports: `"use client"`, `useRouter` from `next/navigation`, `Trash2` from `lucide-react`. No other.
- No `useState`, no error toast — fire-and-forget matches the rest of the app's mutation style (see `src/app/trips/[tripId]/page.tsx`, `handleRemove` for stops).

**Acceptance criteria:**
- TripCard still renders identically on first paint when the delete button is not interacted with.

- The grid row-equalisation on `/trips` is preserved — sibling cards in the same row still match heights.
