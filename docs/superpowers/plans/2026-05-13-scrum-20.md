# SCRUM-20 — Delete Trip button

**Goal:** Add a "delete trip" affordance to each card on the My Trips page that works equally well with touch and mouse, asks for confirmation before firing, and removes the trip *and all of its associated stops* from DynamoDB. After deletion the My Trips list refreshes to reflect the removal.

**Ticket:** https://devotonomy.atlassian.net/browse/SCRUM-20
**Branch:** `scrum-20`

---

## Context for the implementer

**My Trips today.** `src/app/trips/page.tsx` is a server component that fetches the user's trips + stops via ElectroDB, derives the per-trip status/segment data, and renders a grid of `<TripCard>`s. Each `TripCard` (`src/components/trip/TripCard.tsx`) is currently a plain server-rendered component whose root element is `<Link href={`/trips/${tripId}`}>` — clicking anywhere on the card navigates to that trip's detail page. The top of the card is `<h3>{name}</h3>` plus a row of status icons (CheckCircle2 / Clock).

**Existing DELETE route.** `src/app/api/trips/[tripId]/route.ts` already exports a `DELETE` handler (lines 120–131) that calls `TripEntity.delete(...)` and returns `{ success: true }`. **It does not delete the trip's stops** — stops live in a separate single-table partition keyed by `tripId`, and orphan rows would remain after a trip is deleted. The ticket explicitly says "delete all trip information", so we need cascade delete in the API.

**Auth/ownership.** The current handler relies on `TripEntity.delete({ userId: user.id, tripId })` for ownership — DynamoDB's partition key is composed of `userId`, so a delete for a trip the caller doesn't own is a no-op against a missing key. The stops table has no `userId` in its partition key (`pk = tripId`, `sk = stopId`), so deleting stops by `tripId` alone is sufficient *only if* we've first verified the caller owns the trip. The cleanest fix: query the trip by `(userId, tripId)` first, 404 if missing, only then iterate stops + delete trip. This matches the ownership check in the existing `PATCH` handler in the same file (lines 53–59).

**Confirmation precedent.** `src/components/profile/DeleteProfileButton.tsx` uses the browser-native `window.confirm("Delete this profile?")`. It's the only confirmation pattern in the codebase. We'll mirror it — no custom Dialog primitive exists and inventing one for this ticket would balloon the scope. `window.confirm` is reliably touch-friendly and mouse-friendly across iOS Safari, Android Chrome, and desktop browsers.

**Touch + mouse affordance.** The closest existing destructive icon is the stop-row Trash2 in `src/components/stop/StopList.tsx` line 348 — but it's `opacity-0 group-hover:opacity-100`, which means touch users can't see it. The ticket explicitly calls out "touch device AND mouse friendly", so we deliberately diverge from that pattern and make the delete button **always visible** on TripCard. We place it at the top-right of the card (in the same row as the status icons) with a `text-base`-friendly tap target — `h-11 w-11` on mobile, `h-8 w-8` from `lg:` up — matching the StopList delete button's responsive sizing convention.

**Click-through hazard.** Because the whole card is a `<Link>`, the delete button must call `e.preventDefault()` *and* `e.stopPropagation()` on click; otherwise tapping it would both fire the delete handler *and* navigate to the (about-to-be-deleted) trip page. The `confirm()` prompt happens first, but the navigation still races with it on some browsers.

**Refresh strategy.** After a successful DELETE the My Trips page (server component) needs to re-render to drop the trip from the grid. `router.refresh()` from `next/navigation` is the right primitive — it re-runs the server component, re-fetches trips, and patches the DOM. The `DeleteProfileButton` already uses this pattern.

**Validation.** `npx tsc --noEmit && npm run lint && npm run build`. No automated tests in the repo. The dev-stack at `https://192.168.50.155:3100` covers the visual/click-through check during Phase 4d.

**Don't touch unrelated code.** Two files change: the trip route (cascade delete) and `TripCard.tsx` (client conversion + delete button). The trips page itself doesn't change — it already passes `tripId` to TripCard, which is all the button needs.

## Task 1 — Cascade-delete stops in the DELETE route

**Files:** Modify `src/app/api/trips/[tripId]/route.ts`.

- [ ] **Step 1.** Replace the body of the existing `DELETE` handler (lines 120–131). The new flow:
    1. `requireAuth()` → throws → 401 on the outer catch.
    2. Ownership check via `TripEntity.query.byUser({ userId: user.id }).where(... eq(tid, tripId) ...).go()`. If `data[0]` is undefined → 404. Mirrors the PATCH handler at lines 54–59 in the same file.
    3. Query all stops for the trip: `await StopEntity.query.byTrip({ tripId }).go()`. ElectroDB returns `{ data, cursor }` — destructure `.data` (per AGENTS.md).
    4. Delete each stop concurrently with `Promise.all(stops.data.map(s => StopEntity.delete({ tripId, stopId: s.stopId }).go()))`. Concurrent is fine; these are independent items and the table partitions distribute the work.
    5. Delete the trip: `await TripEntity.delete({ userId: user.id, tripId }).go()`.
    6. Return `{ success: true }`.

```ts
export async function DELETE(_req: Request, { params }: { params: Promise<{ tripId: string }> }) {
  try {
    const user = await requireAuth();
    const { tripId } = await params;

    // Verify the caller owns this trip before touching stops, since the
    // stop partition key does not include userId.
    const owned = await TripEntity.query.byUser({ userId: user.id })
      .where(({ tripId: tid }, { eq }) => eq(tid, tripId))
      .go();
    if (!owned.data[0]) {
      return NextResponse.json({ error: "Not found" }, { status: 404 });
    }

    const stops = await StopEntity.query.byTrip({ tripId }).go();
    await Promise.all(
      stops.data.map(s => StopEntity.delete({ tripId, stopId: s.stopId }).go())
    );

    await TripEntity.delete({ userId: user.id, tripId }).go();

    return NextResponse.json({ success: true });
  } catch {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }
}
```

The single comment above the ownership check is the only explanatory comment we add — the *why* (stop partition key has no userId) is the non-obvious bit.

- [ ] **Step 2.** Confirm the existing imports at the top of the file already include `TripEntity` and `StopEntity` (they do — both are used by PATCH and GET). No new imports needed.

## Task 2 — Add the delete button to TripCard

**Files:** Modify `src/components/trip/TripCard.tsx`.

- [ ] **Step 1.** Add `"use client"` as the very first line of the file. TripCard becomes a client component so it can manage delete state and call `useRouter`.

- [ ] **Step 2.** Extend the imports:

```tsx
"use client";
import Link from "next/link";
import { useRouter } from "next/navigation";
import { useState } from "react";
import { Calendar, MapPin, CheckCircle2, Clock, Trash2, Loader2 } from "lucide-react";
import { Card, CardContent } from "@/components/ui/card";
```

`useRouter` is for `router.refresh()` after a successful delete. `useState` tracks the in-flight `deleting` state so we can swap the trash icon for a spinner. `Trash2` is the icon; `Loader2` is the in-flight spinner — both already used elsewhere in the codebase via lucide-react.

- [ ] **Step 3.** Inside the `TripCard` function body, before the `return`, add the router and the deleting state plus the handler:

```tsx
const router = useRouter();
const [deleting, setDeleting] = useState(false);

async function handleDelete(e: React.MouseEvent) {
  e.preventDefault();
  e.stopPropagation();
  if (!confirm(`Delete "${name}"? This will also remove all of its stops and bookings. This cannot be undone.`)) return;
  setDeleting(true);
  const res = await fetch(`/api/trips/${tripId}`, { method: "DELETE" });
  if (res.ok) {
    router.refresh();
  } else {
    setDeleting(false);
    alert("Could not delete this trip. Please try again.");
  }
}
```

Notes:
- `e.preventDefault()` blocks the surrounding `<Link>`'s navigation; `e.stopPropagation()` belt-and-braces against any future ancestor handlers.
- The confirm copy includes the trip name and is explicit about cascade ("its stops and bookings") — matching the ticket's "delete all trip information" intent.
- On success we do **not** `setDeleting(false)` — `router.refresh()` will re-render and this component instance will unmount. Leaving the spinner up bridges the gap.
- On failure we use `alert(...)` because we're already in `confirm`-style native UX and there's no custom toast system. Matches the limited error-surface available in this codebase.

- [ ] **Step 4.** Update the top-row JSX (currently lines 46–57). Add the delete button to the right-hand cluster — adjacent to the status icons, always visible:

```tsx
<div className="flex items-start justify-between gap-2 mb-3">
  <h3 className="font-semibold text-base leading-tight text-gray-900">{name}</h3>
  <div className="flex items-center gap-1.5 shrink-0">
    {statuses.map(s => statusConfig[s] && (
      <span key={s} title={statusConfig[s].label}>
        {statusConfig[s].icon(false)}
      </span>
    ))}
    <button
      type="button"
      onClick={handleDelete}
      disabled={deleting}
      aria-label={`Delete ${name}`}
      className="-mr-1.5 -mt-1.5 inline-flex items-center justify-center h-11 w-11 lg:h-8 lg:w-8 rounded-md text-gray-300 hover:text-red-500 hover:bg-red-50 transition-colors disabled:opacity-50 disabled:pointer-events-none"
    >
      {deleting
        ? <Loader2 className="h-4 w-4 animate-spin" />
        : <Trash2 className="h-4 w-4" />}
    </button>
  </div>
</div>
```

Notes on the patch:
- The `statuses.length > 0` outer conditional that used to wrap the icon cluster is gone — the cluster now always renders because the delete button always renders. The `statuses.map(...)` inside still handles the empty case correctly (yields no children).
- `-mr-1.5 -mt-1.5` lets the 44 px touch target visually nest against the card's edge instead of crowding the title — a common pattern for icon buttons inside padded containers. The padding from `CardContent pt-6` and card's `px-6` (the `Card` default) absorbs the negative margin so nothing escapes the card.
- `h-11 w-11 lg:h-8 lg:w-8` matches the StopList delete button's responsive sizing (CLAUDE.md says `lg` is the **only** mobile↔desktop breakpoint).
- Colors are literal Tailwind classes (`text-gray-300`, `hover:text-red-500`, `hover:bg-red-50`) — no dynamic class lookups, per the Tailwind v4 JIT rule.
- `aria-label` includes the trip name so screen readers announce *which* trip is being deleted.
- `type="button"` is essential — the button lives inside a `<Link>` which renders as an `<a>`; a stray `type="submit"` default isn't a risk here, but being explicit avoids `<button>` defaulting to anything implicit if a future refactor moves this into a `<form>`.

- [ ] **Step 5.** Sanity-check the rest of the file. The trash icon is the only behavioural change; everything else (description, dates, members, segments progress bar) stays exactly as is.

## Acceptance

Done when:

1. Each card on `/trips` shows an always-visible trash icon button in the top-right corner, adjacent to any existing status icons.
2. The button is at least 44×44 px on mobile (touch-friendly) and a compact 32×32 px on `lg:` and above (mouse-friendly).
3. Clicking the button shows a native `confirm()` dialog naming the specific trip; cancelling the dialog does nothing.
4. Confirming the dialog calls `DELETE /api/trips/{tripId}`. The endpoint verifies the caller owns the trip, deletes every stop in that trip, then deletes the trip row, and returns `{ success: true }`.
5. After a successful delete the trip disappears from the My Trips grid (the page re-renders via `router.refresh()`); the trip detail page returns a 404 if visited directly.
6. Clicking the button does not trigger the surrounding card's navigation to the trip detail page.
7. `npx tsc --noEmit && npm run lint && npm run build` all pass on the branch.
