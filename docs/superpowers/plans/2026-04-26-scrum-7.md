# SCRUM-7 — Allow typing dates as numbers in trip form

**Ticket:** https://devotonomy.atlassian.net/browse/SCRUM-7

## Problem

When creating a new trip, the start/end date fields use the browser's native
`<input type="date">`. Users who try to type the date as digits (e.g. `01051979`
for 1 May 1979) find the day-segment of the field repeatedly overwrites itself
and the cursor never advances to the month/year segments. On iOS the field
hijacks input entirely with a wheel picker. The intent of the ticket is to
support **both** modes: keyboard typing (numbers) **and** a calendar picker.

Stop dates already use a custom `DatePicker` component
(`src/components/ui/date-picker.tsx`) but its trigger is a read-only button —
it can't be typed into either.

## Solution

Enhance the existing `DatePicker` to accept free-form typed input alongside the
calendar popup, then swap `TripForm.tsx`'s native date inputs over to it. This
fixes trip creation directly and incidentally lets stop dates be typed too,
which matches the same UX request without code duplication.

### DatePicker changes

- Replace the `<button>` trigger with an `<input type="text" inputMode="numeric">`
  that displays the formatted date (`1 May 1979`) when present and unfocused.
- A small calendar icon button sits inside the input on the right and toggles
  the popup. Clicking the input itself does **not** open the popup — that would
  fight typing.
- On focus the displayed value switches to a raw editable form matching the
  user's preferred input shape (`DD/MM/YYYY`).
- On `blur` or `Enter`, parse the text. Accepted shapes:
  - `DDMMYYYY` (8 digits, no separators) — the case in the ticket
  - `DDMMYY` (6 digits) — heuristic: `<= currentYear%100 + 5` ⇒ 20XX, else 19XX
  - `DD/MM/YYYY`, `DD-MM-YYYY`, `DD.MM.YYYY`, `DD MM YYYY` (any of `/-. `)
  - Same set with 2-digit year
- If the parsed date is invalid OR violates `min`/`max`/`disabledRanges`,
  revert the input to the previous valid display value (no silent commit, no
  toast — just don't accept it).
- If valid, call `onChange(formatISO(date))` exactly as the calendar path does.
- Selecting from the popup keeps the same behaviour (`onChange` + close).
- Keep `placeholder` working when value is empty; default placeholder text
  becomes `"DD/MM/YYYY"` so the typing affordance is obvious. Existing call
  sites that pass `placeholder="Select date"` keep their current text.

### TripForm changes

- Replace each `<Input id="..." type="date" {...register("...Date")} />` with
  a `<DatePicker>` controlled via `useController` from react-hook-form (the
  field needs `value` + `onChange`, not a registered native input).
- Min/max constraint: end date must be on/after start date. The existing
  submit-time guard at line 83 stays as a safety net; additionally pass
  `min={startDate}` to the end-date picker once the start is chosen so the
  popup disables earlier days. (Cheap win — no extra tests needed.)

### What stays unchanged

- `formatISO` / `parseISO` — already correct.
- The popup, `react-day-picker`, the disabled-ranges logic, the portal,
  scroll/resize repositioning.
- StopList / StopSearch — they pass the same props and benefit automatically.

## Files

- `src/components/ui/date-picker.tsx` — main change (new typed-input trigger,
  parser helpers).
- `src/components/trip/TripForm.tsx` — swap two `<Input type="date">` for
  `<DatePicker>`, wire via `useController`.

## Validation

- `npx tsc --noEmit && npm run lint && npm run build` — green.
- Manual on the test env (https://192.168.50.155:3101):
  - Trip create page: type `01051979` in start date → field commits to
    "1 May 1979" on blur.
  - Same field: click calendar icon → popup opens, click a date → commits.
  - Type garbage (`abcd`) → reverts to previous value on blur.
  - Type a valid date earlier than `min` (when min is set) → reverts.
  - End-date popup with start date set: earlier days disabled.
  - Stop date picker on an existing trip's stop list: typing still works,
    `min`/`max`/`disabledRanges` still enforced.
- iOS: input is `text-base` (16 px) so no auto-zoom; `inputMode="numeric"`
  brings up the digit keypad.

## Out of scope

- Internationalising the date format (UK `DD/MM/YYYY` only — matches existing
  display format and the user's example).
- Auto-formatting (inserting `/` mid-typing). Keep it simple: parse on commit.
- Rebuilding the calendar popup.
