# SCRUM-7 — Allow typing date numbers in trip date inputs

## Problem

When creating a new trip, users try to type the date as a number string
(e.g. `01051979` for 1 May 1979). The native HTML5 `<input type="date">`
used in `src/components/trip/TripForm.tsx` (lines 125, 130) doesn't
reliably accept a fast numeric burst — the DD field is repeatedly
overwritten and the cursor never advances to MM and YYYY. Users who
prefer keyboard entry can't enter a date.

The user explicitly wants both styles supported: typing digits AND a
calendar picker.

## Approach

Replace the two native `<input type="date">` instances in `TripForm` with
a new hybrid `DateInput` component that combines:

1. A free-form text input that auto-formats digits as `DD/MM/YYYY` while
   the user types (so typing `01051979` produces `01/05/1979`).
2. A calendar popover button next to it (reuses `react-day-picker`,
   mirroring the existing `src/components/ui/date-picker.tsx`).

The component takes/emits ISO `YYYY-MM-DD` like the native control did,
so the API contract (`/api/trips`) and the form's `endDate < startDate`
comparison are unchanged.

UK display order (`DD/MM/YYYY`) matches the user's example
(`01051979` = 1 May 1979). The current native input also defaults to
locale-aware ordering on UK browsers, so this is the format users
already expect.

## Tasks

### Task 1 — Build `DateInput` component

Create `src/components/ui/date-input.tsx`. Public API:

```ts
interface DateInputProps {
  value?: string;                 // ISO YYYY-MM-DD or ""
  onChange: (value: string) => void;
  min?: string;                   // ISO
  max?: string;                   // ISO
  placeholder?: string;
  className?: string;
  id?: string;
  name?: string;
  onBlur?: () => void;            // for react-hook-form Controller
}
```

Behaviour:

- Internal `text` state holds the displayed `DD/MM/YYYY` string. Init
  from `value` via `isoToDisplay()`.
- `onChange` of the text input strips non-digits, slices to 8, and
  rebuilds `DD`, `DD/MM`, `DD/MM/YYYY` segments — assignment-only, no
  in-place mutation, so backspacing works naturally.
- When the user has typed a complete `DD/MM/YYYY` that parses to a real
  date in range, call the prop `onChange(iso)` with `YYYY-MM-DD`.
- On `blur`: if the text parses, canonicalise the displayed text and
  emit ISO; if empty, emit `""`; otherwise leave the bad text for the
  user to fix and don't emit.
- A `<button>` to the right opens a `DayPicker` popover (portal, fixed
  positioning, click-outside + Escape close — copy the pattern from
  `date-picker.tsx`). Selecting a day fills the text input and emits
  ISO.
- `inputMode="numeric"` and `autoComplete="off"` on the text input.
- Honor `min`/`max` in both the text validator and the calendar
  `disabled` matcher.
- Sync `text` from `value` via an effect so external changes
  (e.g. RHF reset) are reflected.

Helper functions, kept inside the file:

- `isoToDisplay(iso?: string): string`
- `displayToIso(display: string): string | null` — strict
  `^\d{2}/\d{2}/\d{4}$`, validates real calendar date, applies
  min/max bounds.
- `formatTyping(raw: string): string` — digits-only, segmented.

The styling matches existing `Input`/`DatePicker` (h-11, text-base,
emerald focus ring, rounded-lg, gray border) so it slots into the
form visually.

### Task 2 — Wire `DateInput` into `TripForm`

Edit `src/components/trip/TripForm.tsx`:

- Import `Controller` from `react-hook-form` and the new `DateInput`.
- Destructure `control` from `useForm`.
- Replace the two `<Input id="...Date" type="date" {...register("...")} />`
  blocks with `<Controller control={control} name="startDate" render={...}>`
  rendering `<DateInput value={field.value} onChange={field.onChange}
  onBlur={field.onBlur} id="startDate" />`.
- For `endDate`, also pass `min={watch("startDate")}` (add `watch` to
  the destructure) so the calendar disables earlier days. The existing
  imperative `data.endDate < data.startDate` check in `onSubmit` stays
  as a safety net.
- Keep the `defaultValues` flow intact — Controller picks up
  `defaultValues.startDate` / `endDate` automatically.

### Task 3 — Validation

Run `npx tsc --noEmit && npm run lint && npm run build`. All three must
pass. No regressions in any other consumer of `Input` or `DatePicker`
(we're not touching either).

## Out of scope

- Stop-level dates already use the custom `DatePicker` (calendar-only
  trigger button, no typing). The ticket is about trip create, not
  stops, and stops don't use the broken native input. Leave them.
- No new tests — repo has no test suite (per `AGENTS.md`).
- No locale toggle — UK `DD/MM/YYYY` matches the ticket example and
  the existing native-input behaviour on a UK browser.
