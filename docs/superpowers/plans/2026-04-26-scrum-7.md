# SCRUM-7 — Typeable date input

**Ticket:** https://devotonomy.atlassian.net/browse/SCRUM-7

## Problem

When creating a trip, the start/end date fields use the browser's native
`<input type="date">`. On iOS Safari (and to a lesser extent Chrome desktop),
typing a date as an 8-digit string like `01051979` does not work: the DD
segment is segmented and each subsequent digit overwrites the previous one
rather than advancing across MM and YYYY. Users who prefer to type a date
quickly cannot, and are forced to use the spinner picker.

Meanwhile, the stop dates already use a custom `DatePicker` component
(`src/components/ui/date-picker.tsx`) backed by `react-day-picker`. That
component is calendar-popover only — it has no typing affordance at all.
The ticket explicitly says *"some users like to use a date picker some
like to just type in the date numbers"*, so the right fix is one
component that supports both.

## Approach

1. Enhance `DatePicker` so its trigger is a typeable text input (UK
   `DD/MM/YYYY` format, auto-inserting slashes as the user types) plus a
   calendar icon button that opens the existing `DayPicker` popover.
2. Replace the two native `<Input type="date">` in `TripForm.tsx` with
   `DatePicker`, wired through `react-hook-form`'s `Controller`.

`StopSearch` and `StopList` already consume `DatePicker` and gain the
typing support automatically with no call-site change.

## Format choice — UK DD/MM/YYYY

The existing `formatDisplay` helper in `date-picker.tsx` already renders
selected dates UK-style (`1 May 1979`), and the ticket's own example uses
`01051979 = 1st May 1979`. Stick with that. Storage stays ISO
`YYYY-MM-DD` (unchanged).

## Tasks

### Task 1 — Typeable DatePicker

Files: `src/components/ui/date-picker.tsx`

- Replace the single `<button>` trigger with a flex container holding an
  `<input type="text" inputMode="numeric" />` and a small calendar
  `<button>` on the right.
- Add helpers:
  - `formatTyping(raw)` — strip non-digits, cap at 8, insert `/` after
    positions 2 and 5. Drives controlled-input value.
  - `parseTyped(raw)` — match `^(\d{1,2})/(\d{1,2})/(\d{4})$`, validate
    real calendar date (handle Feb 30 etc. via `Date` round-trip), return
    ISO `YYYY-MM-DD` or `null`.
  - `displayFromISO(iso)` — `YYYY-MM-DD → DD/MM/YYYY`.
  - `isInRange(iso, min, max, disabledRanges)` — re-uses the same logic
    that builds the day-picker `disabled` matchers.
- Internal state:
  - `inputText` (string, what the user sees while typing).
  - `focused` (boolean) — sync `inputText ← displayFromISO(value)` only
    when not focused, to avoid clobbering mid-type.
  - `invalid` (boolean) — set to `true` only on blur if `inputText` is
    non-empty and won't parse / is out of range. Reset to `false` on the
    next valid keystroke or external value change.
- Behaviour:
  - On every keystroke: run `formatTyping`, set `inputText`. If it parses
    to a valid in-range ISO, call `props.onChange(iso)`. Otherwise leave
    `value` alone (don't clear committed state mid-type).
  - On blur with non-empty unparseable text: set `invalid=true` and snap
    `inputText` back to `displayFromISO(value)` (or empty if no value),
    so the input always reflects committed state.
  - On calendar icon click: toggle popover.
  - On day-picker selection: update both `value` (via `onChange`) and
    `inputText`, close popover.
- Visual: keep the existing border / focus ring; tint border red when
  `invalid`. The calendar icon stays in the same position.
- Wire `id` to the `<input>` (not the button) so `<Label htmlFor>` from
  `TripForm` lands on the typing field.
- Public props (`value`, `onChange`, `min`, `max`, `disabledRanges`,
  `placeholder`, `className`, `id`) are unchanged — `StopSearch` and
  `StopList` need no edits.

### Task 2 — Use DatePicker in TripForm

Files: `src/components/trip/TripForm.tsx`

- Add `Controller` to the `react-hook-form` import. Add `control` to the
  `useForm` destructure.
- Import `DatePicker` from `@/components/ui/date-picker`.
- Replace each native date input with:
  ```tsx
  <Controller
    name="startDate"
    control={control}
    render={({ field }) => (
      <DatePicker id="startDate" value={field.value} onChange={field.onChange} />
    )}
  />
  ```
  and similarly for `endDate` (passing `min={startDateValue}` once we
  watch it via `useWatch` or `watch`).
- The existing `data.endDate < data.startDate` submit-time check stays;
  it's a belt-and-braces check on top of the new `min`.

  then save — `data.startDate` should be `1979-05-01`.
- Calendar icon still opens the picker; selecting a day fills the input.
- Out-of-range / nonsense text reverts on blur and shows red border.
- Stop arrival date in `StopSearch` also accepts typing now.
