# SCRUM-7 — Allow typing date numbers in trip date fields

## Problem

When creating or editing a trip, the start/end date fields are native HTML
`<input type="date">` (`src/components/trip/TripForm.tsx:125,130`). The user
expected to type a date like `01051979` (1 May 1979) and have it land as
`01/05/1979`, but the native segmented input across browsers (especially
mobile Safari, which is the project's primary target) does not auto-advance
DD → MM → YYYY consistently — each digit overwrites the DD segment.

The ticket also explicitly requests that **both** typing-the-numbers and
calendar-picking remain available.

## Approach

Enhance the existing `DatePicker` (`src/components/ui/date-picker.tsx`) to
support typing alongside its existing calendar popover, then swap the two
native date inputs in `TripForm` for it. The `DatePicker` is already used
for stop arrival dates, so this also gains them the typing affordance for
free without changing their call sites.

Display/edit format: `DD/MM/YYYY` (UK convention — matches the ticket's
"01051979 1st may 1979" example and the rest of the app's UK English copy).
Underlying `value` / `onChange` contract stays ISO `YYYY-MM-DD` so callers,
ElectroDB, and validators don't change.

### DatePicker changes

- Replace the trigger `<button>` with a wrapper containing:
  - A text `<input>` that the user can type into. Auto-formats to
    `DD/MM/YYYY`: digits only, slashes inserted after positions 2 and 5,
    capped at 10 chars.
  - A trailing calendar-icon `<button>` that opens the existing popover.
- Local state mirrors what's in the input as the user types (`text`).
  Whenever it parses to a valid date that satisfies `min`/`max`/
  `disabledRanges`, call `onChange(iso)`. Whenever the field is blurred
  with invalid/incomplete content, surface a small inline error (red text
  underneath, same pattern as TripForm's existing error messages).
- When `value` changes externally (calendar pick, parent reset), reflect
  it back into the text input as `DD/MM/YYYY`.
- Keep `id` wiring on the input itself so `<Label htmlFor>` works.
- Keep all existing props (`min`, `max`, `disabledRanges`, `placeholder`,
  `className`, `id`) and the `onChange(iso)` contract.

### TripForm changes

- Drop the two native `<Input type="date" {...register(...)}>` and
  replace with `<DatePicker>` wired through react-hook-form's
  `Controller`. `endDate` gets `min={startDate}` so the calendar
  enforces order (typed entry will also fail validation if it violates).
- Keep the existing onSubmit cross-field check (`endDate < startDate`) as
  defence in depth.

## Out of scope

- Internationalising the date format. The app is single-locale today;
  hard-coding DD/MM/YYYY matches the rest of the UI.
- Touching the stop-date callers' call sites — they get the typing
  affordance automatically because they already use `DatePicker`.
- Replacing react-day-picker, adding a new dependency, or touching the
  database/API contract (still ISO strings end-to-end).

## Validation

- `npx tsc --noEmit && npm run lint && npm run build` all pass.
- Manual: in the worktree dev env at `https://192.168.50.155:3100`,
  create a trip and type `01051979` into start date — expect it to land
  as `01/05/1979` and persist as `1979-05-01`. Then click the calendar
  icon and pick a date — expect the input to update.
