# SCRUM-7 — Allow typing dates in trip form

## Problem

When a user creates a trip and tries to type the date as digits (e.g. `01051979` for 1 May 1979), the native `<input type="date">` in `src/components/trip/TripForm.tsx` consumes each keystroke against the currently-highlighted segment (DD), so the day field is repeatedly overwritten and the user can't enter the rest of the date by typing.

Some users prefer the date picker; others prefer typing. We need to support both.

## Solution

Replace the native `<input type="date">` with a typing-friendly variant of the existing `DatePicker` at `src/components/ui/date-picker.tsx`. The component currently exposes only a button → calendar popover. We extend it to also accept free-text typing in `DD/MM/YYYY` format with auto-slash insertion, while keeping the calendar trigger via a calendar-icon button.

The storage format remains `YYYY-MM-DD` (unchanged contract for the `value`/`onChange` props), so existing callers in `StopSearch.tsx` and `StopList.tsx` are unaffected and inherit the typing capability for free.

## Tasks

### Task 1 — Augment `DatePicker` to accept typed input

In `src/components/ui/date-picker.tsx`:

- Replace the `<button>` trigger with a flex container holding:
  - A controlled `<input type="text" inputMode="numeric">` for typed entry
  - A small `<button>` with the calendar icon to toggle the popover
- Local state `text: string` holds the displayed text (e.g. `"01/05/1979"`).
- Sync `text` from `value` via `useEffect` only when the input is **not** focused — never overwrite mid-edit.
- On every change:
  - Strip non-digit characters; cap at 8 digits.
  - Re-format incrementally as `DD`, `DD/MM`, or `DD/MM/YYYY`.
  - When 8 digits are present, parse as `DD`/`MM`/`YYYY`, validate calendar-roundtrip (rejects 31/02/2025, etc.), and check it falls inside `min`/`max`/`disabledRanges`. If valid, fire `onChange(YYYY-MM-DD)`.
  - When empty, fire `onChange("")` so the form notices the clear.
- On blur: if `text` doesn't parse to the current `value`, snap `text` back to the canonical formatted `value` (or `""`).
- Keep the existing portal popover, day-picker disabled matchers, document-click / Esc handlers, and reposition logic. Selecting a day in the calendar fires `onChange` and updates `text`.
- Show a red border (`border-red-400`) when the user has typed 8 digits that don't parse or fall in a disabled range — gentle visual hint, no blocking.
- Update default `placeholder` to `"DD/MM/YYYY"` so the format is discoverable.

### Task 2 — Use `DatePicker` in `TripForm`

In `src/components/trip/TripForm.tsx`:

- Pull `Controller` and `control` from `react-hook-form`.
- Import `DatePicker`.
- Replace the two `<Input id="..." type="date" {...register(...)} />` blocks with `<Controller render={({ field }) => <DatePicker id=... value={field.value} onChange={field.onChange} /> } />`.
- Keep the existing zod validation (`startDate`, `endDate` non-empty strings) and the start/end ordering check in `onSubmit` — both already work with the `YYYY-MM-DD` value.

### Task 3 — Drop misleading placeholders in stop pickers

`StopSearch.tsx` and `StopList.tsx` pass `placeholder="Select date"` to `DatePicker`. With typing now supported, the default `"DD/MM/YYYY"` is more honest. Remove the `placeholder` prop from both call sites so the new default applies.

## Validation

```bash
npx tsc --noEmit && npm run lint && npm run build
```

Then a manual smoke test against the LAN test URL: open the New Trip form, type `01051979` into the start date, confirm the input fluently shows `01/05/1979` (no overwriting), confirm the calendar icon opens the popover, confirm selecting a day from the picker also populates the input, and confirm the form submits with the date stored correctly. Repeat for the stop date pickers (Add Stop, Edit Stop) which now inherit the typing capability.

## Out of scope

- Time inputs (`<input type="time">`) — the ticket is specifically about date, and time inputs don't have the same digit-overwrite pain.
- Localization of date format (DD/MM vs MM/DD). The codebase already uses DD-first display in `formatDisplay` (`1 May 1979`), so DD/MM/YYYY typing is consistent. No locale switching for now.
