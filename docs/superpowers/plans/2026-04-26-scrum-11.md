## Task 1 — Add `worklog` subcommand to `bin/lib/jira.sh`

**Files:** Modify `bin/lib/jira.sh`.

**Why first:** Step 2 (`ready-for-review` switch) and the skill changes in Step 3 depend on this subcommand existing.

- [ ] **Step 1: Add `worklog` to the usage block.**

In the `usage()` heredoc, after the `comment` line, add:

```
  jira.sh worklog <KEY> [<body>]
```

- [ ] **Step 2: Add the `worklog)` case before `*)`.**

Insert this block (modelled on the existing `comment)` block but with deterministic ADF construction so we don't ask the LLM to interpret structured JSON):

```bash
  worklog)
    [ $# -ge 1 ] || usage
    key="$1"
    if [ $# -ge 2 ]; then
      body="$2"
    else
      body=$(cat)
    fi

    # Convert the plain-text body into Atlassian Document Format. The
    # worklog endpoint requires `comment` to be ADF JSON, not plain text
    # like the comment endpoint accepts. Each line becomes a paragraph;
    # blank lines become empty paragraphs.
    adf_doc=$(printf '%s' "$body" | jq -Rs '
      split("\n")
      | (if .[-1] == "" then .[:-1] else . end)
      | map(
          if . == "" then {type:"paragraph"}
          else {type:"paragraph", content:[{type:"text", text:.}]}
          end
        )
      | {type:"doc", version:1, content:.}
    ')

    # Worklog API requires `started` (ISO 8601 with timezone) and
    # `timeSpentSeconds`. We log a nominal 60s — these entries are
    # status notes from automation, not real time tracking.
    started=$(date -u +"%Y-%m-%dT%H:%M:%S.000+0000")

    request_body=$(jq -n --arg started "$started" --argjson comment "$adf_doc" \
      '{started:$started, timeSpentSeconds:60, comment:$comment}')

    claude -p --allowed-tools "$ALLOWED_TOOLS" <<EOF
Use the atlassian-jira MCP server to add a worklog entry to issue $key.

Call jira_post with path "/rest/api/3/issue/$key/worklog" and this exact JSON body:
---BODY---
$request_body
---/BODY---

After posting successfully, reply with the single word "OK".
EOF
    ;;
```

Why deterministic ADF in shell rather than asking Claude to convert: the existing `comment` subcommand can hand the LLM a free-form string because the comments REST endpoint accepts plain text via the MCP wrapper. The worklog endpoint requires ADF in the request body — letting the LLM build that JSON is a needless source of malformed-doc bugs.

- [ ] **Step 3: Verify the script still parses.**

```bash
bash -n bin/lib/jira.sh
```

Exit 0 means no syntax errors.

## Task 2 — Switch `ready-for-review` from `comment` to `worklog`

**Files:** Modify `bin/lib/jira.sh` (the `ready-for-review)` block).

**Why:** This is the headline change. The `ready-for-review` body is the canonical "automation kanban-board update" — exactly the content the ticket says belongs in the worklog.

- [ ] **Step 1: Change the body-posting line.**

Find:

```bash
    comment_rc=0
    transition_rc=0
    printf '%s' "$body" | "$0" comment "$key" || comment_rc=$?
    "$0" transition "$key" "In Review" || transition_rc=$?

    if [ "$comment_rc" -eq 0 ] && [ "$transition_rc" -eq 0 ]; then
      echo "OK"
    else
      echo "PARTIAL: comment=$comment_rc transition=$transition_rc" >&2
      exit 1
    fi
```

Replace with:

```bash
    worklog_rc=0
    transition_rc=0
    printf '%s' "$body" | "$0" worklog "$key" || worklog_rc=$?
    "$0" transition "$key" "In Review" || transition_rc=$?

    if [ "$worklog_rc" -eq 0 ] && [ "$transition_rc" -eq 0 ]; then
      echo "OK"
    else
      echo "PARTIAL: worklog=$worklog_rc transition=$transition_rc" >&2
      exit 1
    fi
```

The error label changes from `comment=` to `worklog=` so the failure surface stays accurate.

- [ ] **Step 2: Verify parse.**

```bash
bash -n bin/lib/jira.sh
```
