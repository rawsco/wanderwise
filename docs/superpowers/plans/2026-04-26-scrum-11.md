# SCRUM-11 — Worklog Integration

**Goal:** Route automation-emitted Jira updates to the **Worklog**, not Comments. Comments stay reserved for the human↔agent rework conversation that the brief explicitly carves out.

**Ticket:** https://devotonomy.atlassian.net/browse/SCRUM-11
**Branch:** `scrum-11`

---

## Context for the implementer

**Today (pre-change), every automation-side post lands in the Comments lane:**

| Path | What it posts | Caller |
|---|---|---|
| `bin/lib/jira.sh ready-for-review` | "PR opened: <url>… Local test URLs… Worktree…" | `/ticket-work` Phase 4e |
| `bin/lib/jira.sh comment` (refusal-back) | "Started work but hit ambiguity. Pausing…" | `/ticket-work` refusal-back step |
| `bin/lib/jira.sh comment` (hard failure) | "Automated work failed during <phase>…" | `/ticket-work` Phase 5 |

All of these are **status updates emitted by the automation**, not human↔agent conversation. Per the ticket:

> The automation that updates the Jira kanban board should post these updates to the Worklog, comments should be reserved for work cycling through the process that needs fed back to the agent to rework stuff.

So all three move to Worklog. The `comment` subcommand stays available (someone may still want to post a manual comment via the helper), but no automation path calls it.

**Why Worklog is the right Jira primitive here:**

- Worklog entries are first-class on the issue, visible on the kanban card and in the activity feed, and timestamped.
- They are the *only* writable per-issue lane Jira exposes besides comments — there is no separate "agent activity log" API. Worklog with a tiny `timeSpentSeconds: 60` is the standard pattern teams use for agent/bot status posts.
- Comments stay clean for humans to leave rework feedback in.

**Worklog API shape (Jira Cloud REST v3):**

```
POST /rest/api/3/issue/{key}/worklog
{
  "timeSpentSeconds": 60,
  "comment": {
    "type": "doc",
    "version": 1,
    "content": [
      { "type": "paragraph",
        "content": [{ "type": "text", "text": "first paragraph" }] },
      { "type": "paragraph",
        "content": [{ "type": "text", "text": "second paragraph" }] }
    ]
  }
}
```

The `comment` field on a worklog accepts ADF (Atlassian Document Format) — same rules as a regular issue comment. Plain blank lines in the input become empty paragraphs / paragraph breaks. We'll instruct the helper subagent to do the ADF wrap (one paragraph per source line, blank lines become paragraph breaks), exactly the way the existing `comment` subcommand trusts the MCP server to do for issue comments.

**`timeSpentSeconds: 60`** is a deliberate constant — this is a status post, not a real time-tracking entry. One minute is the standard agent-bot convention, small enough to be ignorable in reports.

**Validation — no test suite, real Jira required.** This codebase has no JS/TS tests for `bin/lib/`. Validation is by direct invocation against a real ticket (the SCRUM-11 ticket itself, since we'll post the ready-for-review status from this run). If the worklog API call works end-to-end during Phase 4e of `/ticket-work`, the helper is correct; if it doesn't, we hard-fail and refuse-back. There is no way to dry-run this against a fixture.

**Bash safety.** `bin/lib/jira.sh` keeps `set -euo pipefail`. Don't drop these.

**Don't reformat unrelated code.** The diff for `bin/lib/jira.sh` should add one new subcommand and change ~3 lines in `ready-for-review`. The diff for the skill should be near-mechanical "comment" → "worklog" in the lines that name automation outbound, plus updates to the headings/PARTIAL message.

## Task 1 — Add `worklog` subcommand to `bin/lib/jira.sh`

**Files:** Modify `bin/lib/jira.sh`.

- [ ] **Step 1.** In the header docstring (lines 4-12), add the new subcommand to the inline help:

```
#   worklog <KEY> [<body>]        Post a worklog entry. Body from $2 or stdin.
```

- [ ] **Step 2.** In the `usage()` heredoc, add:

```
  jira.sh worklog <KEY> [<body>]
```

between the `comment` and `transition` lines so the order matches the docstring.

- [ ] **Step 3.** In the `case "$cmd" in` block, add a new arm. Place it immediately after the `comment)` arm so the two related subcommands sit side-by-side:

```bash
  worklog)
    [ $# -ge 1 ] || usage
    key="$1"
    if [ $# -ge 2 ]; then
      body="$2"
    else
      body=$(cat)
    fi
    claude -p --allowed-tools "$ALLOWED_TOOLS" <<EOF
Use the atlassian-jira MCP server to add a worklog entry to issue $key.

Call jira_post with path "/rest/api/3/issue/$key/worklog" and a JSON body
that has:
  - "timeSpentSeconds": 60
  - "comment": an ADF (Atlassian Document Format) document representing
    the body text below. Wrap each non-blank line of the body as its own
    paragraph node. Blank lines in the body become paragraph breaks
    (i.e. they separate paragraphs but do not produce empty paragraph
    nodes).

Body text (between the markers):
---BODY---
$body
---/BODY---

After posting successfully, reply with the single word "OK".
EOF
    ;;
```

Why explicit ADF instructions and not "just trust the MCP server": the existing `comment` subcommand uses natural language ("add the following comment") because the MCP server's `add_comment` tool wraps strings in ADF for you. Worklog has no equivalent shortcut tool — we go through `jira_post` directly, so the body must be valid ADF on the wire. Hand-rolling the ADF in the prompt makes that unambiguous.

- [ ] **Step 4.** Smoke-test the new subcommand by hand from the worktree. Pick a throwaway test value:

```bash
echo "smoke test from worklog subcommand — please ignore" | bin/lib/jira.sh worklog SCRUM-11
```

Expected: `OK` to stdout, the entry appears under SCRUM-11's Work Log tab in Jira. If the call fails (`OK` not printed, or a non-zero exit), fix before moving to Task 2 — the `ready-for-review` change in Task 2 depends on this working.

If smoke-testing pollutes SCRUM-11's worklog visibly, that's acceptable — Phase 4e of this very run will post a real worklog to SCRUM-11 anyway. The smoke test is just an early guarantee that we don't discover the bug at the very end of the run.

## Task 2 — Switch `ready-for-review` to use `worklog`

**Files:** Modify `bin/lib/jira.sh`.

- [ ] **Step 1.** Inside the `ready-for-review)` arm (around line 163), change the `comment_rc` block. Current code:

```bash
    comment_rc=0
    transition_rc=0
    printf '%s' "$body" | "$0" comment "$key" || comment_rc=$?
    "$0" transition "$key" "In Review" || transition_rc=$?

    if [ "$comment_rc" -eq 0 ] && [ "$transition_rc" -eq 0 ]; then
      echo "OK"
    else
      echo "PARTIAL: comment=$comment_rc transition=$transition_rc" >&2
      exit 1
    fi
```

becomes:

```bash
    worklog_rc=0
    transition_rc=0
    printf '%s' "$body" | "$0" worklog "$key" || worklog_rc=$?
    "$0" transition "$key" "In Review" || transition_rc=$?

    if [ "$worklog_rc" -eq 0 ] && [ "$transition_rc" -eq 0 ]; then
      echo "OK"
    else
      echo "PARTIAL: worklog=$worklog_rc transition=$transition_rc" >&2
      exit 1
    fi
```

That's two identifier renames (`comment_rc` → `worklog_rc`) and one verb change in the call. The body composition above it does not change — same printf, same fields.
