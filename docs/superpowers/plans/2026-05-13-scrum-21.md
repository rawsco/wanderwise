# SCRUM-21 — "In Progress" nav item

Surface a top-level menu item, **In Progress**, that highlights when any of the signed-in user's trips covers today, and greys out otherwise.

## Behaviour

For the signed-in user, a trip is "in progress" if `today` is in `[startDate, endDate]` (inclusive both ends), comparing as ISO date prefixes (`YYYY-MM-DD`).

- **0 in-progress trips** → render the label as a non-clickable greyed `<span>` (`text-gray-300`).
- **1+ in-progress trips** → render as a highlighted `<Link>`, emerald brand colour, with a small pulsing dot:
  - If exactly 1 → link to `/trips/[tripId]` for that trip.
  - If 2+ → link to `/trips` (the existing list).
- Position: **leftmost** item in both desktop and mobile menu (before "My Trips"), since the ticket says "at the top".
- Signed-out users see nothing (the existing nav already short-circuits for them).

## Design

`Navbar` is a client component because of `useSession()` and the mobile-menu toggle state, so server-side data fetch needs a wrapper.

Add a server component `NavbarShell.tsx` next to `Navbar.tsx` that:

1. Reads the session via `getServerSession(authOptions)`.
2. If signed in, queries `TripEntity.query.byUser({ userId }).go()`.
3. Filters for trips where `startDate?.slice(0,10) <= today && today <= endDate?.slice(0,10)`, with `today = new Date().toISOString().slice(0, 10)`.
4. Passes `inProgressCount: number` and `inProgressTripId?: string` to `<Navbar />` as props.

`Navbar` accepts the props and renders the new item in two places (desktop + mobile). Colours are written as **literal class strings** in two branches (no dynamic interpolation) per the Tailwind v4 JIT rule in `CLAUDE.md`.

`src/app/layout.tsx` swaps `<Navbar />` for `<NavbarShell />`.

## Files touched

- **New:** `src/components/layout/NavbarShell.tsx` — server component, queries trips, computes in-progress.
- **Edit:** `src/components/layout/Navbar.tsx` — accept `inProgressCount` + `inProgressTripId` props, render new item in desktop and mobile menus.
- **Edit:** `src/app/layout.tsx` — render `<NavbarShell />` instead of `<Navbar />`.

## Risks / non-issues

- **Per-render DB call on every page.** Acceptable — the existing `/trips` page already does the same query, and the nav lives only on authenticated pages where the data is naturally fresh. Single-table query by GSI is cheap.
- **Timezone.** `today` computed in UTC; trip dates are stored as ISO strings. ISO-prefix lexical compare works correctly because both sides are in the same format. Local-vs-UTC drift is acceptable for a "today" indicator.
- **Dynamic Tailwind classes** — avoided by emitting two separate branches with literal class strings.

## Validation

`npx tsc --noEmit && npm run lint && npm run build` — all three must pass. Manual browser check after dev stack is up: create a trip whose range covers today, refresh, confirm highlighted state; verify a trip in the past greys it back out.
