# SCRUM-7 — Replace native date input on trip form

## Problem

`src/components/trip/TripForm.tsx` uses native `<Input type="date">` for `startDate` and `endDate`. Native date inputs handle keyboard input poorly across browsers:

- iOS Safari opens a wheel picker and disallows typing.
- Chrome/Firefox desktop allow typing but each digit overwrites the current segment unless typed at exactly the right cadence; users typing a continuous string like `01051979` see the DD field repeatedly replaced.
- Visual polish on iOS requires extra `min-w-0 overflow-hidden` wrappers (per CLAUDE.md).

The rest of the app — `StopList.tsx`, `StopSearch.tsx` — already uses the calendar-driven `DatePicker` from `src/components/ui/date-picker.tsx`. TripForm is the only outlier.

## Approach

Replace the two native date inputs in `TripForm.tsx` with the existing `DatePicker`. Bridge the controlled `DatePicker` into `react-hook-form` via `Controller`. While there, set the end-date `min` to the currently selected start date so the calendar visually communicates the existing "end ≥ start" rule.

No new components. No new dependencies. No data-model or API changes.

## Tasks

### 1. Switch TripForm date inputs to DatePicker

- Add `Controller` and `useWatch` (or `watch`) to the existing `react-hook-form` import.
- Add `import { DatePicker } from "@/components/ui/date-picker";`.
- Remove the two `<Input … type="date" {...register(...)} />` blocks.
- Replace each with a `Controller name="startDate" | "endDate"` that renders `<DatePicker value={field.value} onChange={field.onChange} id={...} />`.
- Watch `startDate` and pass it as `min` to the end-date `DatePicker` so the calendar disables earlier days. Keep the existing zod `min(1)` validation and the manual `endDate < startDate` check (defense in depth).
- Preserve `defaultValues` flow: existing data is already YYYY-MM-DD strings, which is what `DatePicker` accepts.
- Keep the surrounding labels, error rendering, and grid layout untouched.

### 2. Remove the now-redundant `Input` import if unused

- After the change, `Input` is still used in `AnchorPicker`, so the import stays. No-op verification.

### 3. Validation

Run from worktree root:

```bash
npx tsc --noEmit && npm run lint && npm run build
```

All three must pass.

### 4. Manual sanity check (described, not blocking)

Reviewer test plan in PR body:

- Open the trip form (new + edit).
- Click each date field → calendar pops up.
- Pick a start date → end-date calendar disables earlier days.
- Submit → existing validation still runs.

## Out of scope

- Building a custom typed-date-entry component (would diverge from the rest of the app).
- Touching stop-level date inputs — they already use `DatePicker`.
- Restyling the `DatePicker` itself.
